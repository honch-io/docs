title: Honch SDKs

Honch SDKs collect product analytics from connected hardware. They run inside firmware, embedded Linux software, MicroPython firmware, Arduino ESP32 sketches, or a mobile relay app, then queue events locally and upload compact binary chunks to Honch Capture.

## Start With The Integration Path

If you are setting up Honch for the first time, read these in order:

1. [Quickstart](/quickstart): choose the right SDK, send one event, and verify the first delivery path.
2. Your SDK guide: follow the install, configure, first event, verify, and troubleshoot steps for your target.
3. [Shared Concepts](/concepts): understand identity, queueing, retries, lifecycle events, and compact upload behavior.
4. [FAQ](/faq): use the failure-mode answers when events do not show up or queues behave unexpectedly.

Do not start by adding every product event. First prove that one event can queue and either upload or remain pending for the expected reason.

## Choose The SDK

| SDK | Status | Use it for | Guide |
| --- | --- | --- | --- |
| ESP-IDF | Available, component version `0.1.0` | ESP32-family products built with ESP-IDF. | [ESP-IDF](/sdks/esp-idf) |
| C/POSIX | Available, Shared Core behavior `0.2.0` | Embedded Linux devices, local development harnesses, and visible queue inspection. | [C/POSIX](/sdks/c-posix) |
| MicroPython | Available, Shared Core behavior `0.2.0` | Firmware that can include the `_honch_core` MicroPython user C module. | [MicroPython](/sdks/micropython) |
| Arduino ESP32 | In progress, library version `0.1.0` | ESP32 Arduino sketches using the current wrapper and explicit flush calls. | [Arduino ESP32](/sdks/arduino) |
| React Native Relay | In progress, package version `0.1.0` | Companion apps that receive relay chunks from offline devices and upload them later. | [React Native Relay](/sdks/react-native-relay) |

Shared Core is the source of truth for common SDK behavior: event semantics, identity, lifecycle events, queue policy, retry classification, compact upload encoding, and relay packetization. Platform ports provide storage, transport, timing, randomness, and platform-specific integration.

## What You Should Verify First

For every SDK, the first useful checkpoint is the same:

| Step | Proof |
| --- | --- |
| SDK initializes | Init returns success or does not raise. |
| Identity exists | Device ID is generated, configured, or persisted. |
| Event queues | A first event is accepted locally. |
| Flush runs | Transport attempts `POST /capture`. |
| Failure is understood | Retryable failures stay pending; permanent rejections drop or dead-letter. |

Once those are true, add product events, identity, sessions, battery, connectivity, GPIO, or relay flows.

## What Every SDK Sends

Each event has an event name, a timestamp, a `distinct_id`, and properties. SDK-owned properties such as `$device_id`, `$device_model`, `$firmware_version`, `$sdk_platform`, `$sdk_version`, and `$environment` are added automatically and win over user-supplied values with the same keys.

SDKs also emit lifecycle events where the platform supports them:

- `$device_boot`
- `$device_shutdown`
- `$firmware_update`
- `$battery_low`
- `$session_start`
- `$session_end`
- `$connectivity_change`

## Upload Contract

SDKs upload compact chunk frames:

```text
POST /capture
Content-Type: application/vnd.honch.chunk
X-Honch-Project-Key: <project_api_key>
X-Honch-Stream-Id: <stream_id>
```

Capture also accepts the same compact chunk format on `/e` and `/chunks`. Use the hosted HTTPS capture endpoint in production; local development endpoints can use HTTP when intentionally configured.
