Honch SDKs collect product analytics from connected hardware. They run in firmware, embedded Linux processes, MicroPython firmware, ESP32 Arduino sketches, or a companion mobile relay app. SDKs queue events locally before upload, but embedded RAM queues are volatile unless the host opts into durable queue storage.

## Start Here

1. Read [Quickstart](/quickstart) to choose an SDK and send one event.
2. Check [SDK Status](/status) for stability, platform support, and verification evidence.
3. Follow the SDK guide for your target.
4. Use [Security](/security), [Production Checklist](/production-checklist), and [Troubleshooting](/troubleshooting) before shipping.

Do not start by adding every product event. First prove that one event queues locally and either uploads or remains pending for the expected reason.

## Choose The SDK

| SDK | Public status | Use it for | Guide |
| --- | --- | --- | --- |
| ESP-IDF | Stable `0.2.0` | ESP32-family firmware built with ESP-IDF. | [ESP-IDF](/sdks/esp-idf) |
| C/POSIX | Stable `0.2.0` | Embedded Linux devices, local harnesses, and visible queue inspection. | [C/POSIX](/sdks/c-posix) |
| MicroPython | Stable `0.2.0` | MicroPython firmware that can include the `_honch_core` user C module. | [MicroPython](/sdks/micropython) |
| Arduino ESP32 | Preview `0.1.0` | ESP32 Arduino sketches with cooperative flushing. | [Arduino ESP32](/sdks/arduino) |
| React Native Relay | Preview `0.1.0` | Companion apps that receive relay chunks from offline devices. | [React Native Relay](/sdks/react-native-relay) |

Stable means the SDK is documented as a supported integration path. Preview means the API is usable for evaluation and controlled pilots, but production rollout should wait for the validation gates listed in the SDK guide.

## What Every SDK Does

Every direct-upload SDK lets you:

- configure a project key and Capture endpoint;
- identify the device and optional analytics identity;
- track typed events with SDK-owned context;
- queue locally before upload;
- flush explicitly or through cooperative tick work;
- preserve retryable failures and stop retrying permanent failures.

Check each SDK guide for durability defaults. ESP-IDF and Arduino ESP32 use caller-owned RAM queues by default; reboot-safe event queues require caller-provided storage ops.

React Native Relay is different: it is not a firmware analytics SDK. It is the mobile side of a BLE/offline relay path for devices that cannot upload directly.

## First Verification Target

For every SDK, verify:

| Step | Proof |
| --- | --- |
| SDK initializes | Init returns success or does not raise. |
| Identity exists | Device ID is configured, generated, or persisted. |
| Event queues | A first event is accepted locally. |
| Flush runs | The SDK attempts `POST /capture`. |
| Failure is understood | Retryable failures stay pending; permanent failures drop or dead-letter. |

Once this path works, add product events, identity, sessions, battery, connectivity, host-owned GPIO tracking, or relay flows.
