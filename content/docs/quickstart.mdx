This guide is the shortest useful path through the SDKs. Pick the runtime your device can support, send one event, then verify that the SDK queued or delivered it before you add product-specific tracking.

## 1. Choose The SDK

| Target | Use | Start here |
| --- | --- | --- |
| ESP32 firmware built with ESP-IDF | ESP-IDF component | [ESP-IDF guide](/sdks/esp-idf) |
| Embedded Linux, POSIX device process, or local C harness | C/POSIX SDK | [C/POSIX guide](/sdks/c-posix) |
| MicroPython firmware with custom user modules | MicroPython SDK plus `_honch_core` | [MicroPython guide](/sdks/micropython) |
| ESP32 Arduino sketches | Arduino ESP32 library | [Arduino guide](/sdks/arduino) |
| BLE/offline device forwarding through a companion app | React Native Relay | [React Native Relay guide](/sdks/react-native-relay) |

If you are not sure, start with C/POSIX on your development machine. It exercises the same shared behavior as the embedded ports without flashing hardware.

## 2. Set Your Capture Values

Every direct-upload SDK needs the same product values:

| Value | Example | Why it matters |
| --- | --- | --- |
| Project API key | `project-key` | Sent as `X-Honch-Project-Key`. |
| Capture endpoint | `https://capture.honch.io` | SDKs append `/capture`. |
| Device model | `demo-board` | Lets you filter by hardware type. |
| Firmware version | `1.0.0` | Drives firmware rollout and update events. |
| Environment | `production` | Defaults to `production` when omitted. |

For local capture testing, use your local service URL as the endpoint or host. Production firmware should use the hosted HTTPS endpoint.

## 3. Send A First Event

### ESP-IDF

Install the component from an ESP-IDF project:

```bash
idf.py add-dependency "honch-io/honch^0.1.0"
```

Initialize NVS before Honch, then track and flush:

```c
#include "honch.h"
#include "nvs_flash.h"

static uint8_t event_buffer[16384];

void app_main(void)
{
    nvs_flash_init();
    // Initialize Wi-Fi and time sync before expecting delivery.

    honch_config_t config = {
        .api_key = CONFIG_HONCH_API_KEY,
        .host = "https://capture.honch.io",
        .device_model = "demo-board",
        .firmware_version = "1.0.0",
        .event_buffer = event_buffer,
        .event_buffer_size = sizeof(event_buffer),
    };

    if (honch_init(&config) == HONCH_OK) {
        honch_track("app_started", "{\"source\":\"esp-idf\"}");
        honch_flush();
    }
}
```

Expected result: init queues lifecycle events, `app_started` is queued, and `honch_flush()` attempts a compact `POST /capture` upload when the device has network connectivity.

### C/POSIX

Build the SDK and examples:

```bash
cd ports/posix
cmake -S . -B build -DHONCH_BUILD_EXAMPLES=ON
cmake --build build
```

Use a writable queue directory:

```c
#include "honch/honch.h"

int main(void)
{
    honch_config_t config = {
        .api_key = "project-key",
        .endpoint_url = "https://capture.honch.io",
        .device_model = "linux-gateway",
        .firmware_version = "1.0.0",
        .queue_directory = ".honch-queue",
        .durability_mode = HONCH_DURABILITY_SYNC_ALWAYS,
    };

    honch_client_t *client = NULL;
    if (honch_init(&client, &config) == HONCH_OK) {
        honch_track(client, "app_started", "{\"source\":\"posix\"}");
        honch_flush(client);
        honch_shutdown(client);
    }
}
```

Expected result: `.honch-queue/state/` contains identity state, retryable failures stay in `.honch-queue/pending/`, and accepted events are consumed.

### MicroPython

Build firmware with the C user module. Installing Python files alone is not enough.

```bash
make -C ports/unix \
  USER_C_MODULES=/path/to/SDK/ports/micropython/usermod/honch/micropython.cmake
```

Then use the Python wrapper:

```python
import honch

client = honch.Honch(
    api_key="project-key",
    endpoint_url="https://capture.honch.io",
    device_model="pico-w",
    firmware_version="1.0.0",
    queue_directory="/honch",
)

client.track("app_started", {"source": "micropython"})
client.flush()
client.shutdown()
```

Expected result: files are created below `/honch/pending`, `/honch/dead`, and `/honch/state` as needed. Background flushing is disabled by the current user module, so call `flush()` explicitly.

### Arduino ESP32

Use this for ESP32 Arduino sketches that call `flush()` explicitly:

```cpp
#include <WiFi.h>
#include <Honch.h>

static uint8_t eventBuffer[8192];

void setup() {
  WiFi.begin("ssid", "password");
  while (WiFi.status() != WL_CONNECTED) delay(250);

  HonchConfig config = {};
  config.apiKey = "project-key";
  config.host = "https://capture.honch.io";
  config.deviceModel = "esp32-devkit";
  config.firmwareVersion = "1.0.0";
  config.environment = "production";
  config.eventBuffer = eventBuffer;
  config.eventBufferSize = sizeof(eventBuffer);
  config.flushIntervalSeconds = 60;
  config.flushEventThreshold = 30;

  if (Honch.begin(config)) {
    Honch.track("app_started", "{\"source\":\"arduino\"}");
    Honch.flush();
  }
}

void loop() {}
```

Expected result: `Honch.begin()` returns `true`, the device ID is derived from the ESP32 eFuse MAC, and queue state is stored through ESP32 `Preferences`.

### React Native Relay

Use relay only when the device cannot upload directly. The firmware sends relay frames over BLE; the mobile app stores completed messages, ACKs durable receipt, and uploads to Capture.

```ts
import { NativeEventEmitter, NativeModules } from "react-native";
import { createMMKV } from "react-native-mmkv";
import {
  createMobileRelay,
  createMmkvRelayStore,
  createRelayNativeBindings,
} from "@honch/react-native-relay";

const nativeModule = NativeModules.HonchReactNativeRelay;
const bindings = createRelayNativeBindings(nativeModule);

const relay = createMobileRelay({
  durableStore: createMmkvRelayStore(createMMKV({ id: "honch-relay" })),
  uploaderConfig: {
    endpointUrl: "https://capture.honch.io",
    projectKey: "project-key",
    relayId: "mobile-relay-1",
    relaySdkPlatform: "react-native",
    relaySdkVersion: "0.1.0",
    streamId: (message) => `relay-${message.deviceId}`,
    messageId: (message) => Number(message.sequence),
  },
  bleNative: bindings.bleNative,
  schedulerNative: bindings.schedulerNative,
  frameEvents: new NativeEventEmitter(nativeModule),
});

relay.subscribeNativeFrames();
```

Expected result: valid relay frames are assembled into durable mobile queue entries. The firmware should only receive an ACK after the completed compact message is durably stored.

## 4. Verify The First Event

Check these in order:

1. The SDK init call succeeds.
2. Your event call returns success or does not raise.
3. Local queue state is created.
4. The configured endpoint is correct and the SDK appends `/capture`.
5. Capture returns `202`, `204`, or another success accepted by that port.

If the first event does not appear, do not add more tracking yet. Fix delivery first, then add identity, sessions, and product events.

## 5. Add The First Useful Product Events

After the first event is verified, add only the events you would debug from:

| Event | When to send |
| --- | --- |
| `app_started` | After SDK init and app startup. |
| `button_pressed` | From a task or GPIO helper, not raw ISR context. |
| `mode_changed` | When the device changes user-visible operating state. |
| `sync_started` / `sync_finished` | Around network or relay sync flows. |
| `error_seen` | When firmware sees an actionable recoverable error. |

Keep property payloads small and stable. Prefer names like `mode`, `reason`, `source`, `duration_ms`, and `count` over ad hoc nested blobs.
