---
title: Shared Concepts
description: The event model, identity, local queues, retries, upload behavior, and relay behavior shared by Honch SDKs.
---

The official SDKs share one contract even when each port integrates with a different platform. Shared Core implements the canonical behavior for event semantics, identity, lifecycle events, queue policy, compact upload encoding, and relay packetization.

## Event Model

Each event contains:

| Field | Meaning |
| --- | --- |
| `event` | Event name, up to 128 characters in the Shared Core SDKs. |
| `distinct_id` | The analytics identity attached to the event. Defaults to the device ID. |
| `timestamp` | Assigned when the SDK tracks the event. |
| `properties` | JSON-compatible custom properties plus SDK-owned automatic properties. |

User properties are accepted as JSON strings in C, ESP-IDF, and Arduino, and as dictionaries or JSON-compatible values in MicroPython. The C/POSIX and MicroPython wrappers fail closed on malformed JSON. SDK-owned properties override user values with the same keys.

## Identity

The SDK tracks two IDs:

| ID | Purpose |
| --- | --- |
| `$device_id` | Stable hardware or generated device identity. Persisted by the SDK. |
| `distinct_id` | Analytics identity. Starts as the device ID and changes when you call `identify`. |

`reset` is a factory-reset-style boundary. It clears queued events, resets session state, resets low-battery state, and restores or rotates identity depending on whether the device ID was explicitly configured.

## Automatic Properties

Every event includes:

| Property | When |
| --- | --- |
| `$device_id` | Always. |
| `$device_model` | Always, from SDK config. |
| `$firmware_version` | Always, from SDK config. |
| `$sdk_platform` | Always, from the port. |
| `$sdk_version` | Always, from the shared SDK behavior version. |
| `$environment` | Always, defaults to `production`. |
| `$session_id` | Only while a session is active. |
| `$battery_level` | Only when a battery callback is configured and returns `0`-`100`. |
| `$wifi_rssi` | Only on embedded SDKs that can read Wi-Fi signal strength. |

The SDK explicitly does not stamp noisy values such as free heap, uptime, or hardware revision on every event.

## Lifecycle Events

| Event | Trigger |
| --- | --- |
| `$device_boot` | Queued during initialization. |
| `$device_shutdown` | Queued during shutdown before the final flush attempt. |
| `$firmware_update` | Queued during initialization when the stored firmware version differs from the configured one. |
| `$battery_low` | Queued once when battery falls below the configured threshold; re-armed after recovery. |
| `$session_start` | Queued by `session_start`. |
| `$session_end` | Queued by `session_end` or before replacing an active session. |
| `$connectivity_change` | Queued by SDKs that expose connectivity state, such as MicroPython's wrapper method or ESP connectivity support. |

Embedded reset reasons use strings such as `power_on`, `software`, `panic`, `watchdog`, `deepsleep`, `brownout`, or `unknown` where the port can provide them.

## Queueing And Durability

All SDKs queue before delivery. The queue contract is:

- accepted events remain pending until Capture accepts them;
- retryable failures keep events queued;
- permanent rejections are dropped or moved to dead-letter storage;
- queue length is bounded;
- when the queue is full, the oldest event is dropped to admit newer data.

Durability depends on the port:

| SDK | Storage behavior |
| --- | --- |
| ESP-IDF | RAM-first queue backed by a caller-provided buffer, with NVS used for overflow and dead-letter handling. RAM-resident events can be lost on reset or power loss. |
| C/POSIX | File-backed `pending/`, `dead/`, and `state/` directories under `queue_directory`; default durability fsyncs queued events. |
| MicroPython | File-backed `pending/`, `dead/`, and `state/` directories under `queue_directory` through the user C module. |
| Arduino ESP32 | Preferences-backed state and queue. |
| React Native Relay | MMKV or custom durable store for completed relay messages and incomplete chunk assemblies. |

## Flushing And Retry

The Shared Core SDKs default to a 60 second flush interval, a 30 event flush threshold, batches of 20 events capped at 50, 1 second initial backoff, and 5 minute max backoff. Retry jitter is plus or minus 25%.

Response handling follows the shared upload policy:

| Response | SDK behavior |
| --- | --- |
| `202` | Non-final chunk stored; send the next frame. |
| `204` or other success accepted by a port | Confirm and remove the queued message. |
| `400`, `413`, `415`, `422` | Drop or dead-letter malformed data. |
| `401` | Treat as auth failure and stop retrying that message. |
| `409`, `408`, `429`, `5xx`, network errors | Retry with backoff. |

## Compact Upload Format

SDKs upload compact chunk frames:

```text
POST /capture
Content-Type: application/vnd.honch.chunk
X-Honch-Project-Key: <project_api_key>
X-Honch-Stream-Id: <stream_id>
```

The project key is a transport credential, not part of the device message body. A single HTTP upload can contain a complete final frame; constrained links can split the same compact message into multiple frames.

You do not need to construct the compact message by hand when using an SDK. Treat it as the SDK-owned upload payload; configure the endpoint and project key, then let the SDK handle encoding, chunking, retries, and validation responses.

## Relay Flow

Relay is for devices that cannot upload directly. Firmware splits a queued compact message into relay chunks. A gateway or companion app:

1. validates relay frame version, payload length, checksum, and ordering;
2. reassembles chunks by source device ID and sequence;
3. stores the complete compact message durably;
4. ACKs the firmware only after durable receipt;
5. uploads the compact message to `/capture` with relay headers.

React Native Relay implements this mobile side for iOS and Android hosts.

## Integration Check

When you integrate an SDK, verify the behavior that matters to your product: one event queues, flush attempts delivery, retryable failures stay pending, and permanent rejections are not retried forever.
