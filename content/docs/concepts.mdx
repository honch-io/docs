---
title: Shared Concepts
description: Event model, identity, queues, retries, lifecycle events, and relay behavior shared by Honch SDKs.
---

Honch SDKs share one product contract across different platforms. Platform SDKs adapt that behavior to local storage, transport, time, TLS, Wi-Fi, BLE, and board constraints.

## Event Model

Each event has:

| Field | Meaning |
| --- | --- |
| `event` | Stable event name. Shared Core SDKs limit names to 128 characters. |
| `distinct_id` | Analytics identity. Defaults to the device ID until you call identify. |
| `timestamp` | Device event time assigned when the event is tracked. |
| `properties` | Typed custom properties plus SDK-owned context. |

C-derived SDKs use typed `honch_property_t` values. MicroPython accepts dictionaries containing supported typed values. User-supplied SDK-owned keys are rejected before queueing.

## Identity

The SDK tracks:

| ID | Purpose |
| --- | --- |
| `$device_id` | Stable device identity, generated, configured, or platform-derived. |
| `distinct_id` | Analytics identity that can change after `identify`. |

Use `identify` for account, installation, or owner attribution. Use `reset` only for a factory-reset-style boundary; it clears queue and session state and restores or rotates identity depending on configuration.

## Automatic Properties

Every event includes:

| Property | When |
| --- | --- |
| `$device_id` | Always. |
| `$device_model` | Always, from SDK config. |
| `$firmware_version` | Always, from SDK config. |
| `$sdk_platform` | Always, from the port. |
| `$sdk_version` | Always, from the SDK behavior version. |
| `$environment` | Always, defaults to `production`. |
| `$session_id` | Only while a session is active. |
| `$battery_level` | Only when battery support is configured and returns `0`-`100`. |
| `$wifi_rssi` | Only on embedded SDKs that can read Wi-Fi signal strength. |

The SDK does not stamp noisy values such as free heap, uptime, or hardware revision on every event by default.

## Lifecycle Events

| Event | Trigger |
| --- | --- |
| `$device_boot` | Queued during initialization. |
| `$device_shutdown` | Queued during shutdown before the bounded final flush attempt. |
| `$firmware_update` | Queued during initialization when the stored firmware version changes. |
| `$battery_low` | Queued once when battery falls below the configured threshold, then re-armed after recovery. |
| `$session_start` | Queued by `session_start`. |
| `$session_end` | Queued by `session_end` or before replacing an active session. |
| `$connectivity_change` | Queued by SDKs that expose explicit connectivity helpers. |

Embedded reset reasons use values such as `power_on`, `software`, `panic`, `watchdog`, `deepsleep`, `brownout`, or `unknown` where the port can provide them.

## Queueing And Persistence

All SDKs queue before delivery:

- accepted events stay pending until Capture accepts them;
- retryable failures stay queued;
- permanent rejections are dropped or dead-lettered according to port capability;
- queues are bounded;
- full queues drop oldest data to admit newer data where the port exposes shared queue policy.

| SDK | Storage behavior |
| --- | --- |
| ESP-IDF | RAM-only event queue by default. Events survive reset or power loss only when the host supplies durable `event_queue_ops`. |
| C/POSIX | File-backed `pending/`, `dead/`, and `state/` directories under `queue_directory`. |
| MicroPython | Caller-provided RAM `event_buffer` through `_honch_core`; events and identity state are volatile unless the application supplies its own durable source for the required `device_id`. |
| Arduino ESP32 | Caller-owned RAM queue by default. Durable queue and state storage are explicit opt-in through caller-provided ops. |
| React Native Relay | MMKV or custom durable mobile store for relay messages and assemblies. |

State persistence and event queue durability are separate concerns on embedded SDKs. State storage preserves identity and firmware-version state. Event queue storage preserves tracked events that have not yet been accepted by Capture.

## Flushing And Retry

The SDK does not hide upload work in an unmanaged background thread. Call the cooperative tick API from a scheduler or low-priority task, or call flush explicitly when you want an immediate drain attempt. On embedded targets, tick and flush can perform network I/O when uploads are enabled; keep them out of ISRs and customer-critical loops.

Default retry behavior starts at 1 second, caps at 5 minutes, and applies plus or minus 25% jitter. Network failures, timeouts, `408`, `409`, `429`, and `5xx` responses are retryable. SDKs that expose connectivity callbacks should avoid upload attempts while offline and return quickly. ESP-IDF also honors HTTP `429` `Retry-After` when Capture provides it.

Direct-upload SDKs may enforce minimum spacing between outbound uploads. ESP-IDF and Arduino ESP32 default to `10000` ms between successful outbound flushes unless the host overrides the setting for benchmark, factory, or explicit high-throughput modes.

Authentication and malformed-data failures are not useful to retry unchanged.

## Relay Flow

Relay is for devices that cannot upload directly. Firmware sends relay chunks to a gateway or companion app. The receiver validates and reassembles chunks, stores the complete message durably, ACKs the device, then uploads the message to Capture.

React Native Relay implements the mobile side of that flow for iOS and Android host apps. It does not replace the direct firmware SDKs for devices with normal HTTP connectivity.
