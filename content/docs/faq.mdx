title: FAQ
description: Common questions and troubleshooting for Honch SDK integrations.
---

## Which SDK should I use?

Use ESP-IDF for ESP32 firmware built with ESP-IDF, C/POSIX for embedded Linux or local C harnesses, MicroPython for firmware that can include `_honch_core`, Arduino ESP32 for current Arduino milestone testing, and React Native Relay for companion apps that forward data from devices without direct internet access.

## Are all SDKs production-ready?

No. ESP-IDF, C/POSIX, and MicroPython have current core-derived implementations. Arduino ESP32 and React Native Relay are in progress and should be treated as milestone packages until hardware and host-app validation are complete.

## What endpoint do SDKs use?

Current SDKs send compact chunk frames to:

```text
POST /capture
Content-Type: application/vnd.honch.chunk
X-Honch-Project-Key: <project_api_key>
X-Honch-Stream-Id: <stream_id>
```

Capture also accepts the same format on `/e` and `/chunks`. The old `/batch` + CBOR upload path is not the active SDK contract.

## Should I use HTTPS?

Use the hosted HTTPS Capture endpoint in production. SDKs send to the URL you configure, so local development can intentionally use HTTP against a local capture service.

## What gets collected automatically?

Every SDK-owned event includes device identity, device model, firmware version, SDK platform, SDK version, and environment. Sessions add `$session_id`. Battery-aware ports can add `$battery_level` and `$battery_low`. Embedded adapters can add connectivity and Wi-Fi signal events or properties where supported.

## Can user properties override SDK properties?

No. SDK-owned auto properties win over user-supplied values with the same keys. Reserved keys such as `$device_id`, `$sdk_platform`, `$sdk_version`, and `$session_id` should be treated as SDK-owned.

## What property format should I pass?

C, ESP-IDF, and Arduino accept JSON strings. C/POSIX and MicroPython validate JSON-shaped public inputs at the API boundary. MicroPython accepts dictionaries and JSON-compatible values, then serializes them before calling `_honch_core`.

## What happens when the device is offline?

Events are queued locally before upload. Retryable transport failures keep events queued. Permanent capture rejections are dropped or moved to dead-letter storage depending on the port.

Durability varies by SDK:

| SDK | Offline storage |
| --- | --- |
| ESP-IDF | RAM-first queue with NVS overflow/dead-letter handling. RAM-resident events can be lost on reset. |
| C/POSIX | File-backed queue under `queue_directory`. |
| MicroPython | File-backed queue under `queue_directory`. |
| Arduino ESP32 | ESP32 `Preferences` queue in the current milestone. |
| React Native Relay | MMKV or custom durable mobile store. |

## Why are events queued but not appearing?

Check in this order:

1. Confirm the SDK initialized successfully.
2. Confirm the endpoint URL includes the intended host and the SDK appends `/capture`.
3. Confirm Wi-Fi or network connectivity.
4. Confirm your API key.
5. Check logs for `401`, retryable transport errors, or local queue pressure.
6. For local development, verify Capture health and that your local stack accepts compact chunks.

## What does HTTP 401 mean?

Capture rejected the project key. SDKs treat this as an auth failure and do not keep retrying the same message indefinitely. Fix the configured API key and retry with new events.

## What errors are retryable?

Network failures, timeouts, rate limits, `409`, and `5xx` responses are retryable. The shared retry policy starts at 1 second, caps at 5 minutes, and adds plus or minus 25% jitter.

## What happens on reset?

`reset` clears queued data, clears session state, resets low-battery state, and restores or rotates identity depending on whether a device ID was explicitly configured. Treat it as a factory-reset boundary, not as a normal session action.

## Why does MicroPython require a user C module?

The MicroPython wrapper deliberately stays thin. The shared C core owns queueing, identity, compact encoding, retry classification, and packetization. Firmware must include `_honch_core`; installing the Python wrapper alone is not enough.

## Can I call `track` from an ISR?

Do not call full SDK tracking APIs from ISR context. ESP-IDF's `honch_track_gpio` is the safe path for GPIO edge tracking: the ISR only queues the pin number and a worker task later calls `honch_track`.

## How many GPIO pins can ESP-IDF track?

The ESP-IDF GPIO adapter supports up to 8 pins and applies 50 ms debounce. It configures pins as inputs with pull-up enabled and emits an event with a `pin` property.

## How does React Native Relay ACK a device?

The mobile relay ACKs a firmware message only after all relay chunks pass validation and the completed compact message is durably stored on mobile. Capture upload success is tracked separately by the relay queue and retry scheduler.

## Where are tests?

Each SDK has focused verification:

| Area | Command |
| --- | --- |
| POSIX | `ctest --test-dir ports/posix/build --output-on-failure` after a POSIX build. |
| ESP-IDF | `python3 ports/esp-idf/tests/test_cbor_migration.py`; firmware builds through `idf.py`. |
| MicroPython | `PYTHONPATH=ports/micropython python3 -m unittest discover -s ports/micropython/tests -t . -v`. |
| Arduino | `ports/arduino/scripts/verify-arduino.sh`. |
| React Native Relay | `bun run typecheck`, `bun run test`, and native verification scripts from `ports/react-native-relay`. |
| Wire fixtures | `python3 -m unittest spec.conformance.test_wire_v2_fixtures`. |
