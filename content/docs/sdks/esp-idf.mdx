The ESP-IDF SDK is a C component for ESP32-family firmware. It wraps the shared C core with ESP-IDF adapters for NVS, HTTP, Wi-Fi metadata, time, randomness, FreeRTOS locking, and GPIO edge tracking.

Status: available as component version `0.1.0`, built on shared SDK behavior version `0.2.0`.

## Requirements

- ESP-IDF `>=5.0`.
- ESP32-family board supported by ESP-IDF.
- NVS initialized before `honch_init`.
- Wi-Fi configured for delivery.
- Honch project API key.

The component depends on ESP-IDF packages for NVS, HTTP client, Wi-Fi, event loop, TLS certificate bundle, timers, pthread support, GPIO, CBOR, and cJSON.

## Install

Use ESP Component Manager:

```bash
idf.py add-dependency "honch-io/honch^0.1.0"
```

Or vendor the SDK repository as a component:

```bash
git submodule add https://github.com/honch-io/honch.git components/honch
```

Manual vendoring must include the whole SDK repository, not only `ports/esp-idf/honch`, because the component compiles shared sources from `core/`.

## Configure

```c
typedef struct {
    const char *api_key;
    const char *host;
    const char *device_model;
    const char *firmware_version;
    const char *environment;
    uint8_t *event_buffer;
    size_t event_buffer_size;
    uint32_t flush_interval_seconds;
    uint32_t flush_event_threshold;
    int (*battery_callback)(void);
    int battery_low_threshold;
} honch_config_t;
```

| Field | Required | Default |
| --- | --- | --- |
| `api_key` | Yes | none |
| `host` | Yes | none |
| `device_model` | Yes | none |
| `firmware_version` | Yes | none |
| `environment` | No | `production` |
| `event_buffer` | Yes | none |
| `event_buffer_size` | Yes | recommend at least `8192` bytes |
| `flush_interval_seconds` | No | `60` |
| `flush_event_threshold` | No | `30` |
| `battery_callback` | No | `NULL` |
| `battery_low_threshold` | No | `15` |

Config strings are copied into fixed-size internal buffers during init. Keep values short and product-readable: device model names, semantic firmware versions, and environment labels are used for filtering analytics later.

## Initialize

```c
#include "honch.h"
#include "nvs_flash.h"

static uint8_t event_buffer[16384];

void app_main(void)
{
    esp_err_t ret = nvs_flash_init();
    if (ret == ESP_ERR_NVS_NO_FREE_PAGES || ret == ESP_ERR_NVS_NEW_VERSION_FOUND) {
        nvs_flash_erase();
        ret = nvs_flash_init();
    }

    // Initialize Wi-Fi and, if available, SNTP before tracking production events.

    honch_config_t config = {
        .api_key = CONFIG_HONCH_API_KEY,
        .host = CONFIG_HONCH_HOST,
        .device_model = "demo-board",
        .firmware_version = "1.0.0",
        .event_buffer = event_buffer,
        .event_buffer_size = sizeof(event_buffer),
    };

    honch_err_t err = honch_init(&config);
    if (err != HONCH_OK) {
        return;
    }

    honch_track("app_started", "{\"source\":\"esp-idf\"}");
    honch_flush();
}
```

`honch_init` starts the shared core, initializes the RAM-first/NVS storage adapter, initializes the HTTP transport, queues boot and firmware lifecycle events, starts background flushing, and starts the GPIO worker task.

## Public API

```c
honch_err_t honch_init(const honch_config_t *config);
honch_err_t honch_shutdown(void);

honch_err_t honch_track(const char *event, const char *properties_json);
honch_err_t honch_identify(const char *distinct_id, const char *properties_json);
honch_err_t honch_set_property(const char *key, const char *value_json);

honch_err_t honch_session_start(const char *session_name);
honch_err_t honch_session_end(void);

honch_err_t honch_flush(void);
honch_err_t honch_reset(void);

const char *honch_get_device_id(void);

honch_err_t honch_track_gpio(gpio_num_t pin, const char *event_name, honch_gpio_mode_t mode);
```

`properties_json` and identity traits are JSON object strings. `value_json` in `honch_set_property` is a JSON value string. Invalid input returns `HONCH_ERR_INVALID_ARG`.

## Events And Identity

- `honch_track` queues a custom event.
- `honch_identify` persists a new `distinct_id` and queues `$identify`.
- `honch_set_property` queues `$set_property`; it does not persist future context.
- `honch_session_start` queues `$session_start` and adds `$session_id` to later events.
- `honch_session_end` queues `$session_end` and clears the active session.
- `honch_reset` clears queued events and resets identity state.

The ESP-IDF adapter stores `device_id`, `distinct_id`, and firmware version in NVS. When no explicit device ID is configured through the core, the current adapter generates and persists a random 32-character hex ID.

## GPIO Tracking

```c
honch_track_gpio(GPIO_NUM_0, "button_pressed", HONCH_GPIO_FALLING_EDGE);
```

Supported modes:

- `HONCH_GPIO_RISING_EDGE`
- `HONCH_GPIO_FALLING_EDGE`
- `HONCH_GPIO_BOTH_EDGES`

The adapter supports up to 8 tracked pins. It configures each pin as input with pull-up enabled, installs an ISR service when needed, queues pin numbers from ISR context, and calls `honch_track` from a FreeRTOS worker task after 50 ms debounce. The emitted event includes `{"pin": <gpio_number>}`.

## Queue And Transport

The default public init path uses a RAM-first queue. `event_buffer` stores RAM queue entries up to 64 events and the available buffer size. When the RAM path cannot accept another event or NVS-backed events already exist, the adapter uses NVS queue storage. NVS also stores dead-lettered events and identity state.

Tradeoff: RAM-resident events are fast but can be lost on reset or power loss. Use an NVS-only custom integration if every queued event must survive power loss.

Flush sends compact chunk frames to:

```text
POST <host>/capture
Content-Type: application/vnd.honch.chunk
X-Honch-Project-Key: <api_key>
X-Honch-Stream-Id: <stream_id>
```

HTTP `202` means a non-final chunk was stored; `204` means the full message was accepted. `401` is treated as auth failure. `408`, `409`, `429`, `5xx`, and network errors are retryable.

## Example App

The example app configures Wi-Fi, SNTP, NVS, Honch, GPIO 0 tracking, sessions, heartbeat tracking, and manual flushes.

```bash
cd ports/esp-idf/example
idf.py menuconfig
idf.py set-target esp32s3
idf.py flash monitor
```

Look for `HONCH_HTTP_PAYLOAD` and transport status logs while testing. If events queue but do not send, verify Wi-Fi, time/TLS setup, capture URL, and API key.

## Verify
