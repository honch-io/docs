The ESP-IDF SDK is the production ESP32 component. It uses Shared Core behavior with ESP-IDF integration for NVS, HTTP, Wi-Fi metadata, time, randomness, FreeRTOS locking, and GPIO edge tracking.

Status: available as component version `0.2.0`, built on shared SDK behavior version `0.2.0`.

## Before You Start

You need:

| Requirement | Notes |
| --- | --- |
| ESP-IDF | Version `>=5.0`. |
| Target board | Any ESP32-family board supported by ESP-IDF. |
| NVS | Must be initialized before `honch_init`. |
| Network | Wi-Fi must be configured before events can reach Capture. |
| Project key | Sent as `X-Honch-Project-Key`. |
| Event buffer | Caller-owned RAM buffer, recommend at least `8192` bytes. |

The component depends on ESP-IDF packages for NVS, HTTP client, Wi-Fi, event loop, TLS certificate bundle, timers, locking, and GPIO. The public event API uses typed properties from the canonical C core.

## 1. Install The Component

Use ESP Component Manager:

```bash
idf.py add-dependency "honch-io/honch^0.2.0"
```

Or vendor the SDK repository:

```bash
git submodule add https://github.com/honch-io/honch.git components/honch
```

Manual vendoring must include the whole SDK repository, not only `ports/esp-idf/honch`, because the component compiles shared sources from `core/`.

Expected result: your ESP-IDF project can include `honch.h`, and the component build can resolve the SDK sources.

## 2. Add Configuration

The public config is:

```c
typedef struct {
    const char *api_key;
    const char *host;
    const char *device_model;
    const char *firmware_version;
    const char *environment;
    uint8_t *event_buffer;
    size_t event_buffer_size;
    uint32_t flush_interval_seconds;
    uint32_t flush_event_threshold;
    int (*battery_callback)(void);
    int battery_low_threshold;
    honch_durability_mode_t durability_mode;
} honch_config_t;
```

Required fields:

| Field | What to put there |
| --- | --- |
| `api_key` | Project API key. |
| `host` | Capture base URL, for example `https://capture.honch.io`. |
| `device_model` | Product or board model, for example `controller-v2`. |
| `firmware_version` | Current firmware version. |
| `event_buffer` | Static or long-lived RAM buffer. |
| `event_buffer_size` | Buffer size in bytes. |

Optional fields:

| Field | Default |
| --- | --- |
| `environment` | `production` |
| `flush_interval_seconds` | `60` |
| `flush_event_threshold` | `30` |
| `battery_callback` | `NULL` |
| `battery_low_threshold` | `15` |
| `durability_mode` | `HONCH_DURABILITY_OS_BUFFERED` |

Config strings are copied into internal buffers during init. Keep values short, stable, and useful for filtering.

## 3. Initialize NVS, Network, And Honch

```c
#include "honch.h"
#include "nvs_flash.h"

static uint8_t event_buffer[16384];

void app_main(void)
{
    esp_err_t ret = nvs_flash_init();
    if (ret == ESP_ERR_NVS_NO_FREE_PAGES || ret == ESP_ERR_NVS_NEW_VERSION_FOUND) {
        nvs_flash_erase();
        ret = nvs_flash_init();
    }
    if (ret != ESP_OK) {
        return;
    }

    // Initialize Wi-Fi and SNTP here.

    honch_config_t config = {
        .api_key = CONFIG_HONCH_API_KEY,
        .host = CONFIG_HONCH_HOST,
        .device_model = "demo-board",
        .firmware_version = "1.0.0",
        .event_buffer = event_buffer,
        .event_buffer_size = sizeof(event_buffer),
    };

    if (honch_init(&config) != HONCH_OK) {
        return;
    }

    const honch_property_t properties[] = {
        honch_prop("source", honch_str("esp-idf")),
    };
    honch_track("app_started", properties, 1);
    honch_tick();
}
```

Expected result after `honch_init`:

- device identity is loaded or generated;
- boot lifecycle events are queued;
- firmware version is compared against persisted state;
- RAM-first/NVS storage is ready;
- HTTP transport is ready;
- cooperative flush state is ready for `honch_tick()`;
- the GPIO worker task starts.

## 4. Track Custom Events

```c
const honch_property_t button_properties[] = {
    honch_prop("pin", honch_i64(0)),
    honch_prop("source", honch_str("boot_button")),
};
honch_track("button_pressed", button_properties, 2);

const honch_property_t mode_properties[] = {
    honch_prop("mode", honch_str("recording")),
};
honch_track("mode_changed", mode_properties, 1);
honch_tick();
```

Rules:

- Properties use typed `honch_property_t` values.
- Invalid typed input returns `HONCH_ERR_INVALID_ARG`.
- SDK-owned properties such as `$device_id`, `$sdk_version`, and `$session_id` are rejected when supplied by users.
- Do not call `honch_track` directly from ISR context.

Expected result: events are queued immediately. Delivery happens on explicit `honch_flush` or through cooperative `honch_tick()` when the flush threshold or interval is reached and network is available.

## 5. Add Identity And Sessions

Use identity when you want analytics to follow an account, installation, or user-facing owner:

```c
const honch_property_t traits[] = {
    honch_prop("plan", honch_str("beta")),
};
honch_identify("account-123", traits, 1);
```

Use sessions to group bounded flows:

```c
honch_session_start("calibration");
const honch_property_t step_properties[] = {
    honch_prop("step", honch_i64(1)),
};
honch_track("calibration_step", step_properties, 1);
honch_session_end();
```

Expected result: identified events use the new `distinct_id`, and events inside a session include `$session_id`.

## 6. Track GPIO Safely

```c
honch_track_gpio(GPIO_NUM_0, "button_pressed", HONCH_GPIO_FALLING_EDGE);
```

Supported modes:

- `HONCH_GPIO_RISING_EDGE`
- `HONCH_GPIO_FALLING_EDGE`
- `HONCH_GPIO_BOTH_EDGES`

The GPIO helper supports up to 8 tracked pins. It configures each pin as input with pull-up enabled, installs an ISR service when needed, queues pin numbers from ISR context, and calls `honch_track` from a FreeRTOS worker task after 50 ms debounce.

Expected event properties:

```json
{"pin":0}
```

## 7. Verify On Hardware

Run the example app:

```bash
cd ports/esp-idf/example
idf.py menuconfig
idf.py set-target esp32s3
idf.py flash monitor
```

Check these signs:

| Sign | Meaning |
| --- | --- |
| Honch init succeeds | Config, NVS, storage, and transport are initialized. |
| Device gets IP | The SDK can attempt uploads. |
| `HONCH_HTTP_PAYLOAD` or transport logs appear | Flush path is running. |
| HTTP `202` or `204` | Capture accepted the chunk state. |
| HTTP `401` | Project key is wrong. |

If events queue but do not upload, verify Wi-Fi, time/TLS setup, capture host, and API key before adding more events.

## Queue And Durability

The default public init path uses a RAM-first queue. `event_buffer` stores RAM queue entries up to 64 events and the available buffer size. When the RAM path cannot accept another event or NVS-backed events already exist, the SDK uses NVS queue storage.

Tradeoff: RAM-resident events are fast but can be lost on reset or power loss. NVS stores identity, firmware version, and overflow events. Permanent capture rejections are consumed instead of being written to an unused dead-letter namespace.

## Transport Contract

Flush sends compact chunk frames to:

```text
POST <host>/capture
Content-Type: application/vnd.honch.chunk
X-Honch-Project-Key: <api_key>
X-Honch-Stream-Id: <stream_id>
```

Response handling:

| Response | Behavior |
| --- | --- |
| `202` | Non-final chunk stored; send the next frame. |
| `204` | Full message accepted. |
| `401` | Auth failure; do not retry the same message indefinitely. |
| `408`, `409`, `429`, `5xx`, network errors | Retry with backoff. |

Queued events use HQR1 typed event records. Flush encodes wire-v2 compact event messages and sends chunk frames to Capture. Promoted context properties are encoded once in message context rather than duplicated into every event property set. Event timestamps represent device event time; boot-relative timestamps are normalized when realtime becomes valid.

## Troubleshooting

| Symptom | Check |
| --- | --- |
| `honch_init` fails | NVS init, required config fields, event buffer pointer and size. |
| Events never send | Wi-Fi IP, TLS time sync, host URL, API key. |
| `401` in logs | Replace the project key and send a new event. |
| GPIO events missing | Pin number, edge mode, pin conflicts, 8-pin limit. |
| Queue grows | Network path or Capture response is retryable; inspect transport logs. |
| Events lost after reset | They were likely still in RAM queue. |
