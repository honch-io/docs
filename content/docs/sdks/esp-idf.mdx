---
title: ESP-IDF
description: Install, configure, verify, and troubleshoot Honch in ESP32 firmware built with ESP-IDF.
---

The ESP-IDF SDK is the stable Honch integration path for ESP32-family firmware built with ESP-IDF. It adapts shared SDK behavior to ESP-IDF storage, HTTP, Wi-Fi metadata, time, randomness, FreeRTOS locking, TLS, and GPIO edge tracking.

Status: Stable `0.2.0`.

## Requirements

| Requirement | Notes |
| --- | --- |
| ESP-IDF | Version `>=5.0`. |
| Target board | ESP32-family board supported by ESP-IDF; record your verified board in product release evidence. |
| NVS | Initialize before `honch_init`. |
| Network | Configure Wi-Fi before expecting uploads. |
| Time/TLS | Configure time and certificate trust for HTTPS. |
| Project key | Sent as `X-Honch-Project-Key`. |
| Event buffer | Caller-owned RAM buffer; at least `8192` bytes recommended. |

## Install

From an ESP-IDF project, use ESP Component Manager:

```bash
idf.py add-dependency "honch-io/honch^0.2.0"
```

Or vendor the SDK repository:

```bash
git submodule add https://github.com/honch-io/honch.git components/honch
```

Manual vendoring must include the whole SDK repository because the component compiles shared sources from `core/`.

## Configure

```c
typedef struct {
    const char *api_key;
    const char *host;
    const char *device_model;
    const char *firmware_version;
    const char *environment;
    uint8_t *event_buffer;
    size_t event_buffer_size;
    uint32_t flush_interval_seconds;
    uint32_t flush_event_threshold;
    int (*battery_callback)(void);
    int battery_low_threshold;
    honch_durability_mode_t durability_mode;
} honch_config_t;
```

Required fields are `api_key`, `host`, `device_model`, `firmware_version`, `event_buffer`, and `event_buffer_size`. Defaults include `environment = "production"`, `flush_interval_seconds = 60`, `flush_event_threshold = 30`, `battery_low_threshold = 15`, and `durability_mode = HONCH_DURABILITY_OS_BUFFERED`.

## First Event

```c
#include "honch.h"
#include "nvs_flash.h"

static uint8_t event_buffer[16384];

void app_main(void)
{
    esp_err_t ret = nvs_flash_init();
    if (ret != ESP_OK) {
        return;
    }

    // Initialize Wi-Fi and SNTP before expecting HTTPS delivery.

    honch_config_t config = {
        .api_key = CONFIG_HONCH_API_KEY,
        .host = "https://capture.honch.io",
        .device_model = "demo-board",
        .firmware_version = "1.0.0",
        .event_buffer = event_buffer,
        .event_buffer_size = sizeof(event_buffer),
    };

    if (honch_init(&config) == HONCH_OK) {
        const honch_property_t properties[] = {
            honch_prop("source", honch_str("esp-idf")),
        };
        honch_track("app_started", properties, 1);
        honch_tick();
    }
}
```

Call `honch_tick()` from your firmware loop or scheduler for cooperative scheduled flushing. Call `honch_flush()` when you want an immediate delivery attempt.

## Identity, Sessions, And GPIO

Use `honch_identify` when events should attach to an account, installation, or user-facing owner. Use sessions to group bounded flows.

```c
const honch_property_t traits[] = {
    honch_prop("plan", honch_str("beta")),
};
honch_identify("account-123", traits, 1);

honch_session_start("calibration");
honch_track("calibration_step", NULL, 0);
honch_session_end();
```

For GPIO events, use the ESP-IDF helper instead of calling `honch_track` from ISR context:

```c
honch_track_gpio(GPIO_NUM_0, "button_pressed", HONCH_GPIO_FALLING_EDGE);
```

The helper supports up to 8 tracked pins, configures inputs with pull-up enabled, queues work from ISR context, and tracks after a 50 ms debounce window.

## Persistence

The public init path uses a RAM-first queue. RAM-resident events are fast but can be lost on reset or power loss. NVS stores identity, firmware version, and overflow events. Permanent Capture rejections are consumed instead of written to an unused dead-letter namespace.

If your product needs stronger reboot durability than RAM-first queueing provides, record that requirement in your production validation and test the behavior on your target.

## Security

Use `https://capture.honch.io` in production. Verify device time and certificate trust before diagnosing event encoding. Do not disable certificate verification in production.

Project keys are credentials. Do not print full keys in logs or send them as event properties.

## Verify

Run the example app:

```bash
cd ports/esp-idf/example
idf.py menuconfig
idf.py set-target esp32s3
idf.py flash monitor
```

Expected signs:

| Sign | Meaning |
| --- | --- |
| Honch init succeeds | Config, NVS, storage, and transport initialized. |
| Device gets IP | Upload can be attempted. |
| Transport logs appear | Flush path is running. |
| HTTP `202` or `204` | Capture accepted the chunk state. |
| HTTP `401` | Project key is wrong. |

## Troubleshooting

| Symptom | Check |
| --- | --- |
| `honch_init` fails | NVS init, required config fields, event buffer pointer and size. |
| Events never send | Wi-Fi IP, TLS time sync, host URL, project key. |
| `401` in logs | Replace the project key and send a new event. |
| GPIO events missing | Pin number, edge mode, pin conflicts, 8-pin limit. |
| Queue grows | Network path or Capture response is retryable. |
| Events lost after reset | They were likely still in RAM queue. |
