title: MicroPython
description: Build firmware with _honch_core, install the wrapper, and verify MicroPython events.
---

The MicroPython SDK is a stable thin Python wrapper around the `_honch_core` user C module. It is not a standalone pure-Python SDK. Firmware must include the Honch user module so shared SDK behavior owns identity, queueing, retry, compact encoding, and packetization.

Status: Stable `0.2.0`.

## Requirements

| Requirement | Notes |
| --- | --- |
| MicroPython checkout | Build firmware from source for the target port. |
| Honch user module | `ports/micropython/usermod/honch/micropython.cmake`. |
| Python wrapper | Freeze through `manifest.py` or install with `mip` after firmware includes `_honch_core`. |
| Queue directory | Writable filesystem path on the device. |
| Network stack | Board-specific; transport imports `urequests`. |
| Project key | Sent as `X-Honch-Project-Key`. |

CircuitPython is not covered by this SDK because it does not use the same MicroPython user C module flow.

## Build Firmware

For the Unix port:

```bash
make -C ports/unix \
  USER_C_MODULES=/path/to/SDK/ports/micropython/usermod/honch/micropython.cmake
```

For board firmware:

```bash
make BOARD=MYBOARD \
  USER_C_MODULES=/path/to/SDK/ports/micropython/usermod/honch/micropython.cmake \
  FROZEN_MANIFEST=/path/to/SDK/ports/micropython/manifest.py
```

Expected result: `import _honch_core` succeeds in MicroPython.

## Install Or Freeze The Wrapper

The wrapper can be frozen with `manifest.py`. `mip` can also install wrapper files from the SDK package metadata, but the firmware must already contain `_honch_core`.

Expected result: `import honch` succeeds.

## Configure

Required options are `api_key`, `endpoint_url`, `device_model`, `firmware_version`, and `queue_directory`.

Useful options:

| Option | Default |
| --- | --- |
| `device_id` | Generated and persisted when omitted. |
| `environment` | `production`. |
| `batch_size` | `20`, capped at `50`. |
| `max_queued_events` | `1000`. |
| `max_event_bytes` | `16384`. |
| `transport_timeout_ms` | `10000`. |
| `flush_interval_seconds` | `60`. |
| `flush_event_threshold` | `30`. |
| `battery_low_threshold` | `15`. |

Python `platform=`, `transport=`, `battery_callback=`, and `auto_properties_callback=` hooks are rejected. Board behavior belongs in the C user module.

## First Event

```python
import honch

client = honch.Honch(
    api_key="project-key",
    endpoint_url="https://capture.honch.io",
    device_model="pico-w",
    firmware_version="1.0.0",
    queue_directory="/honch",
)

client.track("app_started", {"source": "micropython"})
client.flush()
client.shutdown()
```

Call `client.tick()` periodically from the device main loop for scheduled cooperative flush work.

## Identity, Sessions, And Connectivity

```python
client.identify("account-123", {"plan": "beta"})

client.session_start("recording")
client.track("recording_started", {"mode": "hdr"})
client.session_end()

client.connected()
client.disconnected()
client.connectivity_changed(True)
```

Connectivity methods suppress duplicate states and queue `$connectivity_change` only when the state changes.

## Errors

The wrapper maps core statuses into Python exceptions:

| Exception | Meaning |
| --- | --- |
| `InvalidArgumentError` | Bad config, event name, identity, properties, or JSON value. |
| `StorageError` | Storage I/O or queue-full condition. |
| `TransportError` | Transport or timeout failure. |
| `RateLimitedError` | Capture rate limit. |
| `ServerError` | Capture server error. |
| `RejectedError` | Permanent Capture rejection. |
| `NotInitializedError` | Client missing or already shut down. |
| `HonchError` | Generic SDK error. |

## Persistence

Choose a queue directory on a filesystem that survives reboot:

```text
/honch/
  pending/
  dead/
  state/
```

The SDK creates subdirectories and files as needed.

## Security

Use HTTPS in production and ensure the board has the network and time setup needed for certificate verification. Keep project keys out of source control, logs, and event properties.

## Verify

Check these before adding more events:

1. `import _honch_core` succeeds.
2. `import honch` succeeds.
3. Client creation does not raise.
4. Queue files appear below `queue_directory`.
5. `flush()` reaches the endpoint or leaves retryable files pending.

## Troubleshooting

| Symptom | Check |
| --- | --- |
| `ImportError: _honch_core` | Firmware was not built with the Honch user module. |
| `ImportError: honch` | Wrapper files were not frozen or installed. |
| Client creation fails | Required config values or queue directory. |
| Flush does nothing | Network setup, `urequests`, endpoint URL, project key. |
| Files stay in `pending` | Retryable transport or Capture response. |
| Files move to `dead` | Capture permanently rejected the payload. |
| Out-of-memory behavior | Confirm C heap reservation and runtime headroom for the target port. |
