The C/POSIX SDK is the stable Honch integration path for embedded Linux targets, local development harnesses, and visible queue inspection. It uses shared SDK behavior with POSIX file storage and libcurl transport.

Status: Stable `0.2.0`.

## Requirements

| Requirement | Notes |
| --- | --- |
| CMake | Version `3.20+`. |
| C compiler | C11-capable compiler. |
| libcurl | Used by the POSIX transport. |
| Queue directory | Writable by one active SDK client/process at a time. |
| Project key | Sent as `X-Honch-Project-Key`. |

## Build

From the SDK repository:

```bash
cd ports/posix
cmake -S . -B build -DHONCH_BUILD_EXAMPLES=ON
cmake --build build
```

Example binaries are written under `build/example`.

## Configure

Required fields are `api_key`, `endpoint_url`, `device_model`, `firmware_version`, and `queue_directory`.

Common defaults:

| Field | Default |
| --- | --- |
| `device_id` | Generated and persisted when omitted. |
| `environment` | `production`. |
| `batch_size` | `20`, capped at `50`. |
| `max_queued_events` | `1000`. |
| `max_event_bytes` | `16384`. |
| `transport_timeout_ms` | `10000`. |
| `flush_interval_seconds` | `60`. |
| `flush_min_interval_ms` | `10000`; use `HONCH_FLUSH_MIN_INTERVAL_DISABLED_MS` only for benchmarks. |
| `flush_event_threshold` | `30`. |
| `flush_retry_initial_ms` | `1000`. |
| `flush_retry_max_ms` | `300000`. |
| `durability_mode` | `HONCH_DURABILITY_OS_BUFFERED`. |

There is no hidden background flush thread. Call `honch_tick()` from your loop or scheduler and call `honch_flush()` for an immediate drain attempt.

## First Event

```c
#include "honch/honch.h"

int main(void)
{
    honch_config_t config = {
        .api_key = "project-key",
        .endpoint_url = "https://capture.honch.io",
        .device_model = "linux-gateway",
        .firmware_version = "1.0.0",
        .environment = "production",
        .queue_directory = ".honch-queue",
        .durability_mode = HONCH_DURABILITY_OS_BUFFERED,
    };

    honch_client_t *client = NULL;
    if (honch_init(&client, &config) != HONCH_OK) {
        return 1;
    }

    const honch_property_t properties[] = {
        honch_prop("source", honch_str("posix")),
    };
    honch_track(client, "app_started", properties, 1);
    honch_flush(client);
    honch_shutdown(client);
    return 0;
}
```

## Identity, Sessions, And Properties

Use typed `honch_property_t` arrays for event properties and identify traits. Event names must be nonblank and no longer than 128 characters. Distinct IDs must be nonblank and no longer than 256 characters.

```c
const honch_property_t traits[] = {
    honch_prop("plan", honch_str("beta")),
};
honch_identify(client, "account-123", traits, 1);

honch_session_start(client, "recording");
honch_track(client, "recording_started", NULL, 0);
honch_session_end(client);
```

Invalid input returns `HONCH_ERROR_INVALID_ARGUMENT`. Strings and keys must be valid UTF-8, map keys must be unique, non-finite floats are rejected, and SDK-owned keys cannot be supplied by application code.

## Storage

The queue directory layout is:

```text
.honch-queue/
  pending/
  dead/
  state/
    device_id
    distinct_id
    firmware_version
```

`HONCH_DURABILITY_OS_BUFFERED` is the default. It keeps atomic rename behavior but skips per-event fsyncs for lower enqueue latency and reduced flash wear. Set `HONCH_DURABILITY_SYNC_ALWAYS` when power-loss durability is required; it fsyncs each queued event file and the queue directory before `honch_track()` returns.

Use one active client per queue directory.

## Security

Use HTTPS in production. Local HTTP is only for an intentional local Capture service. Keep project keys out of logs and event properties.

## Verify

```bash
./build/example/posix_device/honch_posix_example
./build/example/connected_camera/honch_connected_camera_example
./build/example/posix_gpio/honch_posix_gpio_example
```

Inspect `pending/`, `dead/`, and `state/` under your queue directory to confirm local behavior.

## Troubleshooting

| Symptom | Check |
| --- | --- |
| `honch_init` returns invalid argument | Required config fields, queue directory, durability mode. |
| Queue files remain in `pending` | Endpoint, network, project key, or retryable Capture response. |
| Files move to `dead` | Capture permanently rejected the payload. |
| No state files | Init did not complete or queue directory is not writable. |
| Two processes conflict | Use one active client per queue directory. |
| Events too large | Reduce property payload size or raise `max_event_bytes`. |
