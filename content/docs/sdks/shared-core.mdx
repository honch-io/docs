---
title: Shared C Core
description: Canonical portable SDK behavior used by the C-derived Honch SDKs.
---

The shared C core lives in `core/` in the SDK repository. Product applications normally use a platform port, but the core is the behavior source for ESP-IDF, C/POSIX, MicroPython, and Arduino ESP32.

## What The Core Owns

- public event semantics;
- strict input validation for event names, identities, and JSON-shaped inputs;
- CBOR queue-entry construction before compact upload encoding;
- SDK-owned automatic properties;
- identify, set-property, sessions, reset, boot, shutdown, firmware update, and battery-low events;
- queue limit policy and drop-oldest behavior;
- retry classification and backoff timing;
- compact wire-v2 event batch encoding;
- compact chunk frame generation;
- relay packetizer support.

Ports own platform details: storage, transport, time, randomness, locking, logging, Wi-Fi or BLE state, and any board-specific adapters.

## Public Core API

```c
honch_status_t honch_core_init(honch_client_t **client, const honch_core_config_t *config);
honch_status_t honch_core_track(honch_client_t *client, const char *event_name, const char *properties_json);
honch_status_t honch_core_identify(honch_client_t *client, const char *distinct_id, const char *traits_json);
honch_status_t honch_core_set_property(honch_client_t *client, const char *key, const char *value_json);
honch_status_t honch_core_session_start(honch_client_t *client, const char *session_name);
honch_status_t honch_core_session_end(honch_client_t *client);
honch_status_t honch_core_flush(honch_client_t *client);
honch_status_t honch_core_reset(honch_client_t *client);
honch_status_t honch_core_shutdown(honch_client_t *client);
const char *honch_core_get_device_id(honch_client_t *client);
honch_status_t honch_core_copy_device_id(honch_client_t *client, char *buffer, size_t buffer_size);
const char *honch_status_string(honch_status_t status);
```

## Core Config

Required fields:

| Field | Purpose |
| --- | --- |
| `api_key` | Project credential sent as `X-Honch-Project-Key`. |
| `endpoint_url` | Capture base URL; transports append `/capture`. |
| `device_model` | Hardware or product model name. |
| `firmware_version` | Current firmware/software version. |
| `queue_directory` | Required even when the port maps it to non-filesystem storage. |
| `durability_mode` | `HONCH_DURABILITY_SYNC_ALWAYS` or `HONCH_DURABILITY_OS_BUFFERED`. |

Common optional fields:

| Field | Default |
| --- | --- |
| `device_id` | Generated and persisted by the port when omitted. |
| `environment` | `production`. |
| `sdk_platform` | `c-posix` if the port does not override it. |
| `batch_size` | `20`, capped at `50`. |
| `max_queued_events` | `1000`. |
| `max_event_bytes` | `16384`. |
| `transport_timeout_ms` | `10000`. |
| `flush_interval_seconds` | `60`. |
| `flush_event_threshold` | `30`. |
| `flush_retry_initial_ms` | `1000`. |
| `flush_retry_max_ms` | `300000`. |
| `battery_low_threshold` | `15`. |

`disable_background_flush` keeps events queued until explicit `flush` calls. `battery_callback` can return `0`-`100` or a negative value when unknown. `auto_properties_callback` lets a port add validated adapter-owned properties such as `$wifi_rssi`.

## Status Codes

| Status | Meaning |
| --- | --- |
| `HONCH_STATUS_OK` | Operation succeeded. |
| `HONCH_STATUS_ERROR_INVALID_ARGUMENT` | Bad config, event name, identity, JSON input, or buffer. |
| `HONCH_STATUS_ERROR_OUT_OF_MEMORY` | Allocation failed. |
| `HONCH_STATUS_ERROR_IO` | Storage or platform I/O failed. |
| `HONCH_STATUS_ERROR_TRANSPORT` | Transport failed. |
| `HONCH_STATUS_ERROR_RATE_LIMITED` | Capture returned a retryable rate limit. |
| `HONCH_STATUS_ERROR_SERVER` | Capture returned a retryable server error. |
| `HONCH_STATUS_ERROR_REJECTED` | Capture permanently rejected data. |
| `HONCH_STATUS_ERROR_NOT_INITIALIZED` | Client is missing, closing, or already shut down. |
| `HONCH_STATUS_ERROR_ALREADY_INITIALIZED` | A singleton-style port is already initialized. |
| `HONCH_STATUS_ERROR_QUEUE_FULL` | Storage adapter could not queue the event. |
| `HONCH_STATUS_ERROR_TIMEOUT` | Transport timed out. |
| `HONCH_STATUS_ERROR_INTERNAL` | Unexpected internal failure. |

## Adapter Interfaces

The core depends on three adapter tables:

| Adapter | Required responsibilities |
| --- | --- |
| `honch_platform_ops_t` | `now_ms`, `uptime_ms`, random bytes, locking, unlocking, and logging. |
| `honch_storage_ops_t` | persistent state, queue push, queue peek/read, consume, dead-letter, drop-oldest, clear, and depth. |
| `honch_transport_ops_t` | post one compact chunk frame to Capture and classify the result. |

Adapters should keep SDK behavior platform-neutral. Board-specific GPIO, Wi-Fi, BLE, battery, TLS, or filesystem choices belong in the port, not in the core.

## Relay Packetizer

The packetizer lets firmware expose queued data to gateways or companion apps:

```c
bool honch_core_data_available(honch_client_t *client, uint32_t source_mask);
honch_status_t honch_packetizer_begin(honch_client_t *client, honch_packetizer_t *packetizer, uint32_t source_mask);
honch_status_t honch_packetizer_next(honch_packetizer_t *packetizer, uint8_t *buffer, size_t buffer_size, size_t *out_size, bool *message_complete);
honch_status_t honch_packetizer_confirm(honch_packetizer_t *packetizer);
honch_status_t honch_packetizer_abort(honch_packetizer_t *packetizer);
```

`HONCH_DATA_SOURCE_EVENTS` is the active source. Logs and diagnostics are reserved in the mask but are not active SDK sources yet.

## Verification

Core tests cover packetization, retry policy, storage contract behavior, and compact wire-v2 fixtures. Run from the SDK root or through the port build:

```bash
cmake -S . -B build
cmake --build build
ctest --test-dir build --output-on-failure
python3 -m unittest spec.conformance.test_wire_v2_fixtures
```
