---
title: React Native Relay
description: Receive relay chunks from offline devices, store them durably, and upload compact messages from a mobile app.
---

React Native Relay is not a device analytics SDK. It is the mobile side of an offline relay path: firmware sends relay chunks over BLE, the mobile app validates and reassembles them, stores completed compact messages durably, ACKs the device, and uploads the message to Honch Capture.

Status: in progress, package version `0.1.0`.

## Before You Start

Use relay when the firmware device cannot upload directly. If the device has normal HTTP connectivity, use the firmware SDK instead.

You need:

| Requirement | Notes |
| --- | --- |
| React Native host app | The package does not include a complete host app. |
| Native bridge | Android exposes upload scheduling hooks. BLE is owned by the host app on both platforms. |
| Durable store | MMKV in production, or another store that implements the relay storage contract. |
| Firmware relay frames | Relay chunk frames from firmware. |
| Project key | Used by the mobile uploader. |

## 1. Install The Package

Install the relay package into your React Native host app, then make sure the TypeScript package and native bridge are available to the app build:

```bash
bun add @honch/react-native-relay react-native-mmkv
```

Expected result: your app can import `@honch/react-native-relay`, and the native module is registered in your iOS and Android host projects.

## 2. Confirm The Firmware Frame Contract

Firmware sends relay chunk frames:

| Field | Meaning |
| --- | --- |
| version | Relay protocol version expected by the package. |
| source type | `1` for events. |
| flags | first/final bits. |
| sequence | Big-endian message sequence. |
| offset | Big-endian payload offset. |
| payload length | Big-endian payload byte count. |
| checksum | Frame checksum. |
| payload | Compact message bytes from the firmware queue. |

The relay validates frame version, reserved values, payload length, checksum, source type consistency, offsets, duplicate chunks, and completion before ACKing the device.

Expected result: malformed frames are rejected and do not ACK the device.

## 3. Configure BLE Notifications

Reserved UUIDs until product-assigned UUIDs are finalized:

```text
Service: 484f4e43-482d-5245-4c41-592d53445631
Frame Notify Characteristic: 484f4e43-482d-5245-4c41-592d4652414d
ACK Write Characteristic: 484f4e43-482d-5245-4c41-592d41434b31
```

The relay package does not scan, connect, subscribe to characteristics, emit frame notifications, or write ACKs. Your host BLE stack should pass each complete relay frame to `relay.receiveFrame(...)` and write returned ACK bytes only when provided.

```ts
await relay.receiveFrame(deviceId, frameBytes, {
  acknowledge: async ({ ackBytes }) => {
    await hostBle.writeRelayAck(deviceId, ackBytes);
  },
});
```

Expected result: your React Native host stores complete messages durably before writing ACK bytes to the device.

## 4. Choose Durable Storage

Use MMKV in production:

```ts
import { createMMKV } from "react-native-mmkv";
import { createMmkvRelayStore } from "@honch/react-native-relay";

const durableStore = createMmkvRelayStore(createMMKV({ id: "honch-relay" }));
```

The MMKV store writes relay state under `honch.relay.queue.v1`. It keeps incomplete chunk assemblies and completed messages until Capture accepts them or the relay permanently drops them.

For local development, `createMemoryDurableStore` or `createJsonFileRelayStore` can help you inspect behavior before wiring MMKV.

## 5. Wire The Relay

```ts
import { NativeModules } from "react-native";
import {
  createMobileRelay,
  createRelayNativeBindings,
  createMmkvRelayStore,
} from "@honch/react-native-relay";
import { createMMKV } from "react-native-mmkv";

const nativeModule = NativeModules.HonchReactNativeRelay;
const bindings = createRelayNativeBindings(nativeModule);

const relay = createMobileRelay({
  durableStore: createMmkvRelayStore(createMMKV({ id: "honch-relay" })),
  uploaderConfig: {
    endpointUrl: "https://i.honch.io",
    projectKey: "project-key",
    relayId: "mobile-relay-1",
    relaySdkPlatform: "react-native",
    relaySdkVersion: "0.1.0",
    streamId: (message) => `relay-${message.deviceId}`,
    messageId: (message) => Number(message.sequence),
  },
  schedulerNative: bindings.schedulerNative,
});

hostBle.onRelayFrame(async ({ deviceId, frameBytes }) => {
  await relay.receiveFrame(deviceId, frameBytes, {
    acknowledge: async ({ ackBytes }) => {
      await hostBle.writeRelayAck(deviceId, ackBytes);
    },
  });
});
```

Expected result: frame events are decoded, validated, assembled into completed messages, and stored durably before the device is ACKed.

## 6. Upload To Capture

Completed device messages are uploaded as canonical compact HTTP chunks:

```text
POST <endpointUrl>/capture
Content-Type: application/vnd.honch.chunk
X-Honch-Project-Key: <projectKey>
X-Honch-Stream-Id: <streamId(message)>
X-Honch-Relay-Id: <relayId>
X-Honch-Relay-SDK-Platform: <relaySdkPlatform>
X-Honch-Relay-SDK-Version: <relaySdkVersion>
```

The relay wraps the completed compact device message in the canonical HTTP chunk frame, but it does not rewrite the compact device message body.

Upload outcomes:

| Outcome | Queue action |
| --- | --- |
| Success response | Mark uploaded and remove queue state. |
| `400`, `401`, `404` | Drop and remove queue state. |
| Network error, `408`, `409`, `429`, `5xx`, other non-success | Keep pending and schedule retry. |

Backoff starts at 1 second, caps at 5 minutes, and applies plus or minus 25% jitter.

## 7. Add Native Host Requirements

iOS:

- Add `NSBluetoothAlwaysUsageDescription`.
- Enable `bluetooth-central` background mode if receipt should continue in the background.
- There is no package-owned iOS native module in `0.1.0`; call `drainUploads()` from the host app foreground lifecycle.

Android:

- Request `BLUETOOTH_SCAN` and `BLUETOOTH_CONNECT` at runtime on Android 12+.
- Request `ACCESS_FINE_LOCATION` where required for BLE scanning.
- Include `androidx.work:work-runtime` for scheduled upload drains.
- Register the package through the consuming React Native Android host.

The package directory does not contain a complete iOS or Android host app, so validate BLE receipt, ACK writes, durable storage, and upload scheduling inside your consuming app.

## Troubleshooting

| Symptom | Check |
| --- | --- |
| Device never gets ACK | Frame validation, checksum, offsets, completion, durable store write. |
| Duplicate chunks arrive | The queue should reject or ignore duplicates by device and sequence. |
| Upload retries forever | Endpoint URL, project key, network, Capture response. |
| Message drops | Response was permanent (`400`, `401`, `404`) or frame was malformed. |
| Frame receipt missing | Host BLE permissions, notify subscription, and handoff to `receiveFrame`. |
| Background upload missing | iOS background mode or Android WorkManager setup. |
