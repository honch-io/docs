---
title: React Native Relay
description: Receive relay chunks from offline devices, store them durably, and upload from a mobile app.
---

React Native Relay is a preview package for companion apps. It is not a firmware analytics SDK. Firmware sends relay chunks over BLE, the host app passes the raw notification bytes into the relay package, the package validates and reassembles them, stores completed messages durably, returns ACK bytes, and uploads to Honch Capture.

Status: Preview `0.1.0`. Use it for evaluation or controlled pilots until your consuming iOS and Android host apps pass native build, BLE, persistence, background, and live Capture validation.

## Requirements

| Requirement | Notes |
| --- | --- |
| React Native host app | The package does not include a complete host app. |
| Host-owned BLE flow | The host app scans, connects, subscribes to notifications, and writes ACKs with its own Bluetooth stack. |
| Durable store | MMKV for production, or another store implementing the relay storage contract. |
| Firmware relay frames | Relay chunks from firmware. |
| Project key | Used by the mobile uploader. |

Use relay only when the device cannot upload directly. If the device has normal HTTP connectivity, use a firmware SDK instead.

## Install

Install the package into your React Native host app and ensure native modules are registered by the host project:

```bash
bun add @honch/react-native-relay react-native-mmkv
```

If your app uses another package manager, install the same packages with that manager and run the native dependency installation flow used by your React Native project.

## Firmware Frame Contract

Firmware sends relay chunks containing compact device message bytes. The relay validates frame version, reserved values, payload length, checksum, source type consistency, offsets, duplicate chunks, and completion before ACKing the device.

Malformed frames must not ACK the device.

## Host-Owned BLE Flow

The relay package does not provide production BLE scan/connect/subscribe code. This is deliberate: many host apps already own a Bluetooth connection to their device, and the relay should plug into that flow instead of adding a second BLE stack.

The host app must:

- scan and connect to the device using the app's existing BLE code;
- subscribe to the Honch relay frame notification characteristic;
- pass each notification payload to `relay.receiveFrame(deviceId, frameBytes, ...)`;
- write the returned ACK bytes to the Honch relay ACK characteristic;
- call `drainUploads()` when the phone has network access.

The ACK is not a Capture upload ACK. It only means the completed relay message was validated and durably stored on the phone.

## Durable Storage

Use MMKV in production:

```ts
import { createMMKV } from "react-native-mmkv";
import { createMmkvRelayStore } from "@honch/react-native-relay";

const durableStore = createMmkvRelayStore(createMMKV({ id: "honch-relay" }));
```

The MMKV store writes per-chunk and per-message records with a small index. By default, it retains up to `4096` chunks and `1024` completed messages for seven days, then drops the oldest or expired entries to bound offline growth. Override those limits when the host app has a smaller storage budget:

```ts
const durableStore = createMmkvRelayStore(createMMKV({ id: "honch-relay" }), {
  maxChunks: 1024,
  maxCompleteMessages: 256,
  ttlMs: 3 * 24 * 60 * 60 * 1000,
});
```

Completed messages and incomplete assemblies remain pending across app restarts until Capture accepts or permanently rejects them, unless they age out or configured queue bounds require dropping oldest state.

## Wire The Relay

```ts
import {
  createMobileRelay,
  createMmkvRelayStore,
} from "@honch/react-native-relay";
import { createMMKV } from "react-native-mmkv";

const relay = createMobileRelay({
  durableStore: createMmkvRelayStore(createMMKV({ id: "honch-relay" })),
  uploaderConfig: {
    endpointUrl: "https://capture.honch.io",
    projectKey: "project-key",
    relayId: "mobile-relay-1",
    relaySdkPlatform: "react-native",
    relaySdkVersion: "0.1.0",
    streamId: (message) => `relay-${message.deviceId}`,
    messageId: (message) => Number(message.sequence),
  },
});

hostBle.onRelayFrame(async ({ deviceId, frameBytes }) => {
  await relay.receiveFrame(deviceId, frameBytes, {
    acknowledge: async ({ ackBytes }) => {
      await hostBle.writeRelayAck(deviceId, ackBytes);
    },
  });
});
```

Expected result: frame bytes are decoded, validated, assembled into completed messages, and stored durably before ACK bytes are returned to the host BLE layer.

`deviceId` should be stable for the active BLE peripheral/connection. It is used to isolate in-progress assemblies and completed message state.

For Android scheduled drains, the SDK still exposes a native upload scheduler binding. Use it only for upload scheduling, not BLE:

```ts
import { NativeModules } from "react-native";
import { createRelayNativeBindings } from "@honch/react-native-relay";

const schedulerNative = createRelayNativeBindings(
  NativeModules.HonchReactNativeRelay,
).schedulerNative;

const relay = createMobileRelay({
  durableStore,
  uploaderConfig,
  schedulerNative,
});
```

## Upload

Completed device messages upload to:

```text
POST <endpointUrl>/capture
Content-Type: application/vnd.honch.chunk
X-Honch-Project-Key: <projectKey>
X-Honch-Stream-Id: <streamId(message)>
X-Honch-Relay-Id: <relayId>
X-Honch-Relay-SDK-Platform: <relaySdkPlatform>
X-Honch-Relay-SDK-Version: <relaySdkVersion>
```

The relay may re-chunk for HTTP, but it must not rewrite the compact device message body.

| Outcome | Queue action |
| --- | --- |
| Success response | Mark uploaded and remove queue state. |
| `400`, `401`, `404` | Drop and remove queue state. |
| Network error, `408`, `409`, `429`, `5xx`, other non-success | Keep pending and schedule retry. |

## Native Host Requirements

iOS:

- Add the Bluetooth usage strings required by the host app's BLE implementation.
- Enable `bluetooth-central` background mode if the host app should continue BLE receipt in the background.
- iOS upload scheduling is foreground-only. Call `drainUploads()` from the host app foreground lifecycle.

Android:

- Request the Bluetooth permissions required by the host app's BLE implementation.
- The relay package does not merge BLE, location, or notification permissions into the host manifest.
- Include `androidx.work:work-runtime` for scheduled upload drains.
- Register the package through the consuming Android host app.
- Register a headless JS task named `HonchRelayUpload` that calls the same durable queue drain path used by foreground upload drains.

```ts
import { AppRegistry } from "react-native";

AppRegistry.registerHeadlessTask("HonchRelayUpload", () => async () => {
  await relay.drainUploads();
});
```

## Security

Use HTTPS in production. Treat project keys and relay IDs as sensitive operational configuration. Do not store relay queue data in a volatile development-only store for production.

## Production Validation

Before production, validate inside the consuming host app:

- native iOS and Android builds;
- host-owned BLE scan, connect, notification, and ACK writes;
- malformed frames rejected without ACK;
- completed messages ACKed only after durable storage;
- MMKV queue state preserved across app restart;
- retryable Capture failures preserved for later upload;
- accepted Capture responses remove queue state exactly once.

## Troubleshooting

| Symptom | Check |
| --- | --- |
| Device never gets ACK | Frame validation, checksum, offsets, completion, durable store write. |
| Duplicate chunks arrive | The queue should reject or ignore duplicates by device and sequence. |
| Upload retries forever | Endpoint URL, project key, network, Capture response. |
| Message drops | Response was permanent or the frame was malformed. |
| Device never appears | Host BLE scan filters, permissions, radio state, or device advertising. |
| Background upload missing | iOS foreground lifecycle drain or Android WorkManager setup. |
